# LLM 驱动决策与行为“去特化”设计方案 (LLM-Driven Cognitive Decision Design)

本文档整理了针对 Hobbs Cafe 咖啡服务场景中发生的“智能体自己等待自己”死锁问题的根源分析，并基于项目**三大铁律**提出了将硬编码逻辑重构为“大模型自主决策”的去特化方案。

---

## 0. 核心架构哲学：全要素虚拟现实与大模型认知解耦

本系统本质上是一个**模拟全要素的虚拟现实世界 (Full-Element Virtual Reality Simulator)**。在这一设计哲学下，系统各层次的职责被严格划分：

1. **信息聚合（Perception / Sensor）**：系统底层只负责实时监测和整理小镇的“全要素信息”（包括物理空间网格、物品交互状态、角色生理特征如饱食度等、联想/空间记忆，以及**当前时刻、星期、绝对时间及动作所剩时长等时间要素**），并将这些物理、生理和时间维度的客观现状组装成自然语言提示词（Prompt）。
2. **认知计算（LLM Brain）**：系统本身不具备任何“社会学意识”或“社交潜规则”。所有的行为动机、协作方案、社交应答、求生抉择，均由大模型（LLM）作为“大脑”在接收到提示词后，结合**当前时间上下文与时限要求**，自主进行逻辑推演并输出行动决策应答。
3. **物理执行（Execution / World Engine）**：系统在获取大模型的决策响应后，由底层引擎控制智能体执行对应动作（如寻路、物品扣减、坐标移动）。
4. **客观规律维护（World Physics Constraints & Time Decay）**：系统代码的硬编码职责**应当且仅应当**作为客观物理底座，去维护这个虚拟世界的“客观物理与时间规律”（例如：A* 寻路与碰撞阻挡、**以时间/步数为驱动的饱食度与精力值代谢衰减**、生命值扣减、库存资源加减等），而不干涉任何属于“行为规范”、“社会习惯”或“社交协议”的范畴（例如：强制顾客就餐前必须等待服务员端茶等）。

## 1. 发生死锁的问题分析

在模拟过程中，曾出现以下死锁日志：
```text
=== [物理拦截器] Isabella Rodriguez 动作被挂起，等待 Isabella 服务咖啡 ===
```

### 1.1 根源分析
在 `execute.py` 的物理层实现中，硬编码了一个 `PHYSICAL DEPENDENCY INTERCEPTOR`（物理依赖拦截器）：
*   当检测到智能体在咖啡馆（Hobbs Cafe）里进行消费行为（如 `drinking coffee`, `having lunch` 等）时，系统会强制检查该就餐位置是否已经触发了 `"served"`（已服务）物理事件。
*   如果该位置尚未有服务事件发生，拦截器会强行挂起智能体的当前动作，使角色进入 `"waiting for coffee/food to be served (waiting for Isabella)"` 状态。
*   然而，**Isabella Rodriguez** 本人既是店里的服务员，又是就餐的智能体。当她在店里吃午餐时，被物理拦截器强行挂起等待自己服务自己，导致自身动作永久阻塞，进入死锁循环。

### 1.2 铁律违背点
该设计违背了项目架构的**铁律 3（消除行为与社交逻辑的硬编码 / 去特化）**：
*   **硬编码社交规则**：像“顾客进食前必须等待服务员端茶送水”这类行为，属于人类社会的社会学/社交协议，决不能在物理代码层写死，而应当由大模型通过感知环境状态后自主规划。
*   **硬编码名字特化**：拦截器中直接绑定了特定店员名字（`waiting for Isabella`），在扩展多咖啡店或店员变更时会直接失效。

## 2. 决策前提示词设计：当前 vs 方案设计

### 2.1 当前项目实际整理给大模型的提示词 (Current Prompt)
在当前的生理危机求生决策（ReAct loop）中，发送给大模型的 Prompt 主要组装了以下基础数据：
*   **身份集 (ISS)**：包含基础人物性格、日常安排等静态文本。
*   **生理状态数值**：Satiety (饱食度), Stamina (精力), Health (生命值) 等 0-100 的粗数值。
*   **背包物品**：Inventory 中的食物列表。
*   **周边物件列表**：仅提供附近有哪些资源物件的名称（例如 `refrigerator`, `cafe customer seating`），不包含微观状态。

#### 当前 Prompt 示例（以 Isabella 为例）：
```text
Name: Isabella Rodriguez
Age: 34
Innate traits: friendly, outgoing, hospitable
Learned traits: Isabella Rodriguez is a cafe owner of Hobbs Cafe.
...
Current Status:
- Satiety (0-100): 20.0
- Stamina (0-100): 75.0
- Health (0-100): 100.0
- Inventory: empty

Nearby Elements & Resources:
cafe customer seating, behind the cafe counter, coffee maker, refrigerator

Task: What is the next survival action for Isabella? 
Choose one action from: "Gather", "Consume", "Rest", or "Idle".
```
*缺陷*：大模型既不知道挨饿的具体后果，也不知道周边餐桌上到底有没有咖啡，更不知道旁边有没有顾客在等她服务，无法做出精细决策，必须依赖底层代码做物理打断。

---

### 2.2 方案设计的新提示词 (Proposed Prompt)
新的设计方案提倡**彻底取消物理拦截器的强行挂起**，转而将**量化生理规则**、**就餐区微观状态**以及**他人的协同预期**作为上下文输入给大模型，由其自主决策。

#### 新方案 Prompt 结构与设计：
```text
你是一个在虚拟小镇运行的智能体决策大脑。

【你的身份与社会记忆】
- 名字: Isabella Rodriguez
- 职业与特质: 你是 Hobbs Cafe 的店主。你负责日常运营咖啡馆、操作咖啡机（coffee maker）制作咖啡并服务顾客。

【你的生理状态与代谢规则】
- 饱食度: 20/100 (警告：你感到极度饥饿！如果饱食度归 0，你的生命值将每步扣减 2.0。)
- 精力值: 75/100
- 你的背包(Inventory): {} (当前为空，没有可以立刻食用的食物)
- 世界客观物理规律: 
  * 执行 "Gather" 动作可以从苹果树(apple_tree)或店内的冰箱(refrigerator)获取食物放入背包。
  * 执行 "Consume" 动作（消耗背包内的食物）能直接回复 +40 饱食度和 +5 生命值。

【你的位置与周边环境状态】
- 当前位置: Hobbs Cafe (咖啡厅柜台后)
- 咖啡厅内的关键物件状态:
  * 冰箱 (refrigerator): 存有食物（苹果）。
  * 咖啡机 (coffee maker): 空闲。
  * 顾客就餐区 (cafe customer seating): 状态为【未提供咖啡】。

【社会协作与他人期望】
- 咖啡厅就餐区有以下事件正在发生:
  * Klaus Mueller 正坐在顾客就餐区，当前动作状态是【等待伊莎贝拉端上咖啡】。

请分析你当下的生理状态与职业职责，自主决定下一步行动。
你可以选择以下行动之一：
1. "Gather" 目标 "refrigerator"：从咖啡馆冰箱中获取食物（以解决你的饥饿危机）。
2. "Brew" / "Serve" 目标 "Klaus Mueller"：操作咖啡机为 Klaus Mueller 制作并端上咖啡。
3. "Idle"：在柜台后闲置。

请给出你的决策 JSON 以及你的内心思考过程（Reasoning）。
```

## 3. 大模型自主决策与路由逻辑 (LLM Autonomous Decision Routing)

获得上述 Prompt 后，智能体会由 LLM 指引在每一步灵活选择行为分支，彻底消除死锁：

1.  **路径一（生理危机优先）**：
    *   Isabella 发现自己极度饥饿（饱食度 20）且背包无食物，但知道冰箱有苹果。
    *   大模型决策：先执行 `Gather` 冰箱，拿到苹果；下一步决策选择 `Consume` 苹果以恢复饱食度。
    *   解决完自身危机后，大模型感知到 Klaus 仍在等待，下达决策执行 `Brew/Serve` 完成店员职责。
2.  **路径二（服务优先 / 社交责任感）**：
    *   若大模型通过性格特质（hospitable, friendly）判定当前 20 的饱食度还能支撑（未归零），应优先满足顾客需求。
    *   大模型决策：优先走向咖啡机执行 `Brew`，然后把咖啡端给 Klaus，服务完毕后再去解决自己的进食问题。
3.  **顾客侧的自适应**：
    *   对于顾客 Klaus，当他走向咖啡桌想要 `"drinking coffee"` 时，由于物理拦截器已被移除，他不会被底层强行挂起。
    *   但大模型感知到桌上的状态是【未提供咖啡】，大模型会主动做出 `Idle` 或 `Wait` 决策（如“坐在座位上等待店员端来咖啡”），若等待过久，他甚至可以自主选择“换一家店”或“直接回家”。

通过上述设计，所有的行为抉择全部收归大模型大脑（铁律 1），而底层的代码只做物理到货、背包扣减与数值结算等“物理底座”的客观规律维护（铁律 2），从而在架构上实现完美的去特化。
